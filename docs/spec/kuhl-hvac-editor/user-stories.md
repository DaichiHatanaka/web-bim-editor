## エピック1: プロジェクト基盤（Phase 0）

### ストーリー 1.1: 空調プロジェクトの作成 🔵

**信頼性**: 🔵 *設計文書 §1.2 ワークフロー・ユーザヒアリング*

**私は** 空調設備設計者 **として**
**新しい空調設計プロジェクトを作成し、建物情報（Plant → Building → Level）を登録したい**
**そうすることで** 空調設計の出発点となるプロジェクト構造を準備できる

**関連要件**: REQ-004, REQ-007, REQ-008

**詳細シナリオ**:
1. エディタを開き、新規プロジェクトを作成する
2. 建物名称、階数、各階の階高を入力する
3. Plant → Building → Level の階層構造が自動生成される
4. ゾーニングフェーズに遷移できる状態になる

**前提条件**:
- ユーザーはログイン済み
- @kuhl/core のスキーマが定義済み

**優先度**: Must Have

---

### ストーリー 1.2: 建築IFCの読込・表示 🔵

**信頼性**: 🔵 *設計文書 §5.1, §1.2 ステップ1*

**私は** 空調設備設計者 **として**
**建築設計者から受け取ったIFCファイルを読み込み、3Dビューに建築躯体を表示したい**
**そうすることで** 建築図面をもとに空調設計を進められる

**関連要件**: REQ-006, REQ-106, REQ-107, REQ-108

**詳細シナリオ**:
1. IFCファイル（建築躯体）をアップロードする
2. web-ifc（WASM）でブラウザ内パースが実行される
3. 壁・床・梁・柱・天井が3Dビューに表示される
4. 階高・天井高が自動取得され、Level情報に反映される
5. ArchitectureRefNodeとして保存される（表示のみ、編集不可）

**前提条件**:
- IFCファイルが正しい形式（IFC 2x3 or IFC 4）

**制約事項**:
- 建築躯体は表示のみ（編集不可）
- 100MB以下のファイルに対応

**優先度**: Must Have

## エピック2: ゾーニング＋負荷概算（Phase 1）

### ストーリー 2.1: 空調ゾーンの描画 🔵

**信頼性**: 🔵 *設計文書 §1.2 ステップ2, §4.1*

**私は** 空調設備設計者 **として**
**建築躯体の上に空調ゾーン（事務室、会議室、サーバー室等）の境界を描画したい**
**そうすることで** 各室の空調負荷を計算するための区分けができる

**関連要件**: REQ-101, REQ-102

**詳細シナリオ**:
1. ゾーニングフェーズで「ゾーン描画」ツールを選択する
2. 建築平面上でクリックしてポリゴン頂点を指定する
3. ポリゴンが半透明着色でリアルタイムプレビューされる
4. Enterで確定、ゾーン名・用途・設計条件を入力する
5. HvacZoneNodeが作成され、ゾーン一覧に追加される

**前提条件**:
- Level が選択されている
- 建築躯体が表示されている（推奨だが必須ではない）

**優先度**: Must Have

---

### ストーリー 2.2: 空調負荷の自動概算 🔵

**信頼性**: 🔵 *設計文書 §3.2*

**私は** 空調設備設計者 **として**
**ゾーンを定義したら、自動的に冷暖房負荷の概算値を確認したい**
**そうすることで** 機器選定の根拠となる負荷データをすぐに得られる

**関連要件**: REQ-103, REQ-104, REQ-105, REQ-110

**詳細シナリオ**:
1. ゾーンの属性（面積、用途、人数、照明密度等）を入力する
2. 外気条件（地域別設計外気温）を設定する
3. システムが自動でm²単価法により冷暖房負荷を算出する
4. 算出結果（冷却負荷、暖房負荷、顕熱比、必要風量、内訳）がゾーン一覧パネルに表示される
5. ゾーン属性を変更すると、自動的に再計算される

**前提条件**:
- HvacZoneが作成済み
- ゾーンの用途と面積が設定済み

**制約事項**:
- m²単価法による概算（将来的にピーク負荷計算に拡張予定）

**優先度**: Must Have

---

### ストーリー 2.3: ゾーン属性の編集 🔵

**信頼性**: 🔵 *設計文書 §2.2 HvacZoneNode*

**私は** 空調設備設計者 **として**
**ゾーンの用途・設計条件・在室人数などを自由に編集したい**
**そうすることで** 設計条件の変更に応じて負荷計算を更新できる

**関連要件**: REQ-102, REQ-110

**詳細シナリオ**:
1. ゾーンを選択し、右パネルのプロパティエディタを開く
2. 用途（office, meeting, server_room等）を変更する
3. 設計条件（室内温湿度、換気量等）を調整する
4. 変更が保存され、負荷計算が自動更新される

**優先度**: Must Have

## エピック3: 機器配置（Phase 2）

### ストーリー 3.1: 空調機器の配置 🔵

**信頼性**: 🔵 *設計文書 §6 Phase 2, §4.1*

**私は** 空調設備設計者 **として**
**AHU、PAC、FCU、制気口などの空調機器を3D空間に配置したい**
**そうすることで** 機器レイアウトを3Dで視覚的に確認できる

**関連要件**: REQ-201, REQ-202, REQ-206

**詳細シナリオ**:
1. 機器フェーズで配置ツール（AHU/PAC/FCU/制気口）を選択する
2. 3Dビュー上でクリックして配置位置を指定する
3. 機器がLOD100（ボックス）またはLOD200（簡易3D）で表示される
4. タグ名（例: "AHU-101"）がラベルとして表示される
5. ドラッグで位置調整、回転ハンドルで向き調整ができる

**前提条件**:
- Levelが選択されている
- HvacZoneが定義済み（推奨）

**優先度**: Must Have

---

### ストーリー 3.2: 機器の諸元入力 🔵

**信頼性**: 🔵 *設計文書 §2.2 AhuNode, PacNode等*

**私は** 空調設備設計者 **として**
**配置した機器の性能諸元（冷却能力、風量、静圧等）を入力・編集したい**
**そうすることで** 機器選定の妥当性を確認し、設計図書に反映できる

**関連要件**: REQ-204, REQ-207

**詳細シナリオ**:
1. 機器を選択し、右パネルの諸元表を開く
2. AHUの場合: 冷却能力(kW)、暖房能力(kW)、風量(m³/h)、機外静圧(Pa)、モーター出力(kW)等を入力
3. PACの場合: 冷却能力(kW)、暖房能力(kW)、COP、冷媒種類等を入力
4. コイル仕様（入口水温、出口水温、水量）を入力
5. 系統（systemId）を指定し、系統ツリーに反映される

**前提条件**:
- 機器が配置済み

**優先度**: Must Have

---

### ストーリー 3.3: 負荷→機器容量の自動マッチング 🔵

**信頼性**: 🔵 *設計文書 §6 Phase 2*

**私は** 空調設備設計者 **として**
**ゾーンの負荷計算結果から、適切な機器容量を自動提案してほしい**
**そうすることで** 機器選定の効率を上げ、過大・過小設計を防げる

**関連要件**: REQ-205

**詳細シナリオ**:
1. ゾーンの負荷計算が完了している状態で、機器配置ツールを使う
2. ゾーンに機器を配置しようとすると、必要容量が表示される
3. 例: 「冷却負荷 50kW → 推奨: AHU 56kW級」
4. ユーザーは提案を受け入れるか、手動で変更できる

**前提条件**:
- ゾーンの負荷計算が完了済み

**制約事項**:
- MVP段階では汎用的な容量提案のみ（メーカーカタログ連携は後回し）

**優先度**: Should Have

---

### ストーリー 3.4: ポートの可視化と接続管理 🔵

**信頼性**: 🔵 *設計文書 §2.2 PortDef*

**私は** 空調設備設計者 **として**
**機器の接続ポート（給気口、冷水入口等）を3Dで確認し、ダクト/配管との接続状態を管理したい**
**そうすることで** 接続漏れや誤接続を防げる

**関連要件**: REQ-203

**詳細シナリオ**:
1. 機器を選択すると、ポート位置にマーカーが表示される
2. 各ポートにはラベル（"給気口", "冷水入口"等）と方向（in/out）が表示される
3. 接続済みポートは緑、未接続ポートは赤で色分けされる
4. ポートをクリックすると接続先情報が表示される

**前提条件**:
- 機器が配置済み

**優先度**: Must Have

---

### ストーリー 3.5: 系統ツリーの管理 🔵

**信頼性**: 🔵 *設計文書 §4.2 左パネル [系統]*

**私は** 空調設備設計者 **として**
**冷水系統、ダクト系統などの系統ツリーを管理し、機器やダクト/配管を系統別に表示・フィルタリングしたい**
**そうすることで** 複雑な設備配置を系統別に整理・確認できる

**関連要件**: REQ-207

**詳細シナリオ**:
1. 左パネルの系統ツリーで系統を作成・編集する
2. 機器やダクト/配管を系統に紐付ける
3. 系統を選択すると、該当系統の要素のみハイライト表示される
4. 系統別にフィルタリングして表示/非表示を切り替えられる

**前提条件**:
- SystemNodeが定義済み

**優先度**: Should Have

## エピック4: ダクトルーティング（Phase 3 — 後続リリース）

### ストーリー 4.1: ダクトの手動ルーティング 🔵

**信頼性**: 🔵 *設計文書 §4.3, §6 Phase 3*

**私は** 空調設備設計者 **として**
**AHUの給気ポートから制気口まで、ダクトのルートを手動で描画したい**
**そうすることで** 天井裏のダクト経路を設計できる

**関連要件**: REQ-301, REQ-302

**詳細シナリオ**:
1. ルートフェーズで「ダクトルート」ツールを選択する
2. 用途（給気/還気/外気/排気）と系統を選択する
3. AHUの給気ポートをクリック → 始点確定
4. 天井裏レベルで中間点をクリック → リアルタイムプレビュー
5. 制気口をクリック → 終点確定、Enter で確定
6. DuctSegmentNode + DuctFittingNodeが自動生成される

**優先度**: Must Have（Phase 3）

---

### ストーリー 4.2: ダクト寸法の自動選定 🔵

**信頼性**: 🔵 *設計文書 §3.3*

**私は** 空調設備設計者 **として**
**ダクトルーティング時に、風量に応じた適切な断面寸法を自動選定してほしい**
**そうすることで** 手計算の手間を省き、規格サイズに自動で合わせられる

**関連要件**: REQ-303

**詳細シナリオ**:
1. ダクトルートを描画する
2. 分岐点での風量按分が自動計算される
3. 等速法（または等圧法）で断面寸法が自動選定される
4. 矩形ダクトの場合、アスペクト比制約（上限4）内で最適なW×Hが選ばれる
5. 規格サイズ（100, 150, 200, 250, 300, ...mm）にスナップされる

**優先度**: Must Have（Phase 3）

---

### ストーリー 4.3: 半自動ルーティング 🔵

**信頼性**: 🔵 *ユーザヒアリング: 半自動*

**私は** 空調設備設計者 **として**
**始点と終点を指定するだけで、システムが最適な経路を提案してほしい**
**そうすることで** ルーティング作業の効率を大幅に上げられる

**関連要件**: REQ-304

**詳細シナリオ**:
1. 始点（AHUポート）と終点（制気口）を指定する
2. システムが建築躯体を考慮した経路を自動提案する
3. 提案経路がプレビュー表示される
4. ユーザーは中間点をドラッグして調整できる
5. 確定するとダクトセグメントが生成される

**優先度**: Should Have（Phase 3）

## エピック6: 検証＋数量拾い（Phase 5 — 後続リリース）

### ストーリー 6.1: 数量拾い出し 🔵

**信頼性**: 🔵 *設計文書 §3.6*

**私は** 空調設備設計者 **として**
**設計した空調設備の数量を自動集計し、積算システムに出力したい**
**そうすることで** 手作業での拾い出し作業を大幅に削減できる

**関連要件**: REQ-501, REQ-502

**詳細シナリオ**:
1. 「拾い」フェーズで「拾い出し実行」を実行する
2. ダクト面積(m²)、配管長さ(m)、機器台数が自動集計される
3. 保温材、ダクト付属品、配管継手、バルブ等も自動拾い
4. 下パネルに集計結果が表示される
5. CSV/Excel（みつもりくん形式）/Rebro形式でエクスポートする

**優先度**: Must Have（Phase 5）
