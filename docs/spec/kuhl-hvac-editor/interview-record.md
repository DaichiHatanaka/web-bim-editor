# Kühl HVAC Editor ヒアリング記録

**作成日**: 2026-03-23
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存の設計文書（`docs/空調基本設計BIM開発.md`）と既存コードベース（Pascal Editor V2）の分析結果をもとに、不明点・曖昧な部分を明確化するためのヒアリングを実施しました。

---

## 質問と回答

### Q1: フォーク戦略 — 既存建築機能の扱い

**質問日時**: 2026-03-23
**カテゴリ**: 既存設計確認
**背景**: 設計文書に「フォーク＆建築ノード削除」とあるが、既存のPascal Editorには14種類の建築ノード、8つのシステム、17のレンダラー、25+のツールが実装済み。削除範囲と移行方針の確認が必要。

**回答**: **完全削除** — 建築ノード・システム・レンダラー・ツールを全て削除し、空調専用に再構成。ArchitectureRefでIFC表示のみ。

**信頼性への影響**:
- REQ-001（Phase 0: スキーマ再構成）の信頼性レベルが 🟡 → 🔵 に向上
- 既存コードの大部分を削除し、共通基盤（BaseNode, useScene, event bus等）のみ残すことが確定

---

### Q2: パッケージ構成

**質問日時**: 2026-03-23
**カテゴリ**: 既存設計確認
**背景**: 設計文書では `@kuhl/core`, `@kuhl/viewer` と記載。既存の `@pascal-app/*` を拡張するか新規スコープにするかで、依存関係とビルド設定が大きく変わる。

**回答**: **新規スコープ @kuhl/** — 新しいパッケージスコープで作成。既存の共通ユーティリティは共有。

**信頼性への影響**:
- パッケージ構成に関するすべての要件の信頼性が 🟡 → 🔵 に向上
- `@kuhl/core`, `@kuhl/viewer` で確定

---

### Q3: ターゲットユーザー

**質問日時**: 2026-03-23
**カテゴリ**: 未定義部分詳細化
**背景**: 設計文書にターゲットユーザーの明記がなく、UI/UXの粒度やエラーメッセージの詳細度に影響。

**回答**: **設備設計者（社内）** — 自社の空調設備設計担当者が使用する社内ツール。

**信頼性への影響**:
- UI/UX関連要件で「専門用語をそのまま使用可」「初心者向けガイド不要」と判断可能
- 新規要件: 社内ユーザー向けの認証・アクセス管理で十分（信頼性: 🔵）

### Q5: 機器データソース（N-BOM・カタログ連携）

**質問日時**: 2026-03-23
**カテゴリ**: 未定義部分詳細化
**背景**: 設計文書にメーカー別カタログパネルやN-BOM連携が記載されているが、データソースが不明。

### Q6: IFCバックエンド

- フォーク方針: 完全削除→空調専用再構成
