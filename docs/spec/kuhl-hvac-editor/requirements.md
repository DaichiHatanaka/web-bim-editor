# Kühl HVAC Editor 要件定義書

## 概要

**Kühl** /kyːl/ は、空調設備の基本設計フェーズに特化したWebベースの3D BIMエディタである。既存のPascal Editor V2の共通基盤（BaseNode, useScene, event bus, React Three Fiber）を活用しつつ、建築ノードを完全に削除し、空調設備専用に再構成する。

**ターゲットユーザー**: 社内の空調設備設計担当者
**技術基盤**: React Three Fiber + WebGPU + Next.js 16 + Zustand + Zod
**パッケージスコープ**: `@kuhl/core`, `@kuhl/viewer`

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: 設計文書・ユーザヒアリングを参考にした確実な要件

### Phase 0: 骨格準備

#### 通常要件

- **REQ-001**: システムは既存の建築ノード（Wall, Slab, Ceiling, Roof, RoofSegment, Door, Window, Item, Zone, Guide, Scan）とそれに関連するシステム・レンダラー・ツールを全て削除しなければならない 🔵 *ユーザヒアリング: 完全削除確定*
- **REQ-002**: システムは `@kuhl/core` パッケージとして、空調設備用のZodスキーマ（HvacZone, AHU, PAC, FCU, VrfOutdoor, VrfIndoor, Diffuser, Damper, Fan, Pump, Chiller, Boiler, CoolingTower, DuctSegment, DuctFitting, PipeSegment, PipeFitting, Valve, System, Support, ArchitectureRef）を定義しなければならない 🔵 *設計文書 §2.1-2.3*
- **REQ-003**: システムは `@kuhl/viewer` パッケージとして、3Dビューポートと空調ノード用レンダラーを提供しなければならない 🔵 *ユーザヒアリング: 新規スコープ @kuhl/*
- **REQ-004**: システムは空間階層（Plant → Building → Level → HvacZone → 機器/ダクト/配管）をフラットノード辞書で管理しなければならない 🔵 *設計文書 §2.1, 既存アーキテクチャ*
- **REQ-005**: システムは `AnyNode` ユニオンを空調ノードタイプで再構成しなければならない 🔵 *設計文書 §2.3*
- **REQ-006**: システムはArchitectureRefNodeとして、IFC建築躯体データを表示のみ（編集不可）で参照できなければならない 🔵 *設計文書 §2.1, §5.1*
- **REQ-007**: システムは既存のBaseNode, useScene, event bus, sceneRegistry, dirty trackingの共通基盤を維持しなければならない 🔵 *既存アーキテクチャ・ユーザヒアリング*
- **REQ-008**: システムはエディタのフェーズを `zone` | `equip` | `route` | `calc` | `takeoff` に再定義しなければならない 🔵 *設計文書 §4.1*

#### 制約要件

- **REQ-009**: `@kuhl/viewer` は `apps/editor` からインポートしてはならない（Viewer Isolation原則） 🔵 *既存CLAUDE.md*
- **REQ-010**: すべてのノードは `.parse()` を使って生成しなければならない 🔵 *既存CLAUDE.md*
- **REQ-011**: システムは既存の認証（better-auth）とデータ保存（Supabase + IndexedDB）の仕組みを使用しなければならない 🔵 *ユーザヒアリング*

### Phase 1: ゾーニング＋負荷概算

#### 通常要件

- **REQ-101**: システムはHvacZoneRendererとして、床面ポリゴンを半透明着色で表示しなければならない 🔵 *設計文書 §6 Phase 1*
- **REQ-102**: システムはZoneDrawToolとして、ユーザーがゾーン境界ポリゴンを描画できる機能を提供しなければならない 🔵 *設計文書 §6 Phase 1*
- **REQ-103**: システムは空調負荷概算（m²単価法）として、ゾーン別の冷暖房負荷を自動算出しなければならない 🔵 *設計文書 §3.2*
- **REQ-104**: 負荷概算は用途別負荷原単位（W/m²）を使用し、方位補正・ガラス面積補正を適用しなければならない 🔵 *設計文書 §3.2 calculateZoneLoad()*
- **REQ-105**: システムはゾーン一覧パネルに、各ゾーンの用途・面積・冷暖房負荷の集計を表示しなければならない 🔵 *設計文書 §6 Phase 1*
- **REQ-106**: システムはIFC建築躯体ファイルを読み込み、壁・床・梁・柱・天井を表示しなければならない 🔵 *設計文書 §5.1*
- **REQ-107**: IFC読込はweb-ifc（WASM）を使用し、ブラウザ内でパースしなければならない 🔵 *設計文書 §5.1*

#### 条件付き要件

- **REQ-108**: IFC読込時に階高・天井高を自動取得できる場合、システムはLevel情報に反映しなければならない 🟡 *設計文書 §5.1から推測*
- **REQ-109**: ゾーンが外壁に面している場合、システムは外壁方位と窓面積比率を負荷計算に反映しなければならない 🔵 *設計文書 §3.2*

#### 状態要件

- **REQ-110**: ゾーンの属性（面積、用途、人数等）が変更された状態にある場合、システムは負荷計算結果を自動的に再計算しなければならない 🔵 *設計文書 §3.2, dirty tracking*

### Phase 2: 機器配置

#### 通常要件

- **REQ-201**: システムはEquipmentRendererとして、LOD100（ボックス）とLOD200（簡易3D形状）の両方で機器を表示しなければならない 🔵 *ユーザヒアリング: LOD100+200*
- **REQ-202**: システムはAHU, PAC, FCU, 制気口（Diffuser）の配置ツール（PlaceTool）を提供しなければならない 🔵 *設計文書 §6 Phase 2*
- **REQ-203**: 各機器はポート定義（給気口、還気口、冷水入口、冷水出口等）を持ち、接続口を可視化しなければならない 🔵 *設計文書 §2.2 PortDef*
- **REQ-204**: システムは諸元表パネル（右パネル）として、選択した機器の性能諸元を表示・編集できなければならない 🔵 *設計文書 §4.2*
- **REQ-205**: システムは負荷計算結果から機器容量の自動マッチング提案を表示しなければならない 🔵 *設計文書 §6 Phase 2*
- **REQ-206**: 各機器はタグ名（例: "AHU-101"）と機器名称を持ち、3Dビュー上にラベル表示できなければならない 🔵 *設計文書 §2.2 HvacEquipmentBase*
- **REQ-207**: 機器は系統（systemId）に紐付けられ、系統別に表示・フィルタリングできなければならない 🔵 *設計文書 §2.2*

#### 条件付き要件

- **REQ-208**: 機器のLODが200に設定されている場合、システムはプロシージャルに簡易3D形状を生成しなければならない 🟡 *ユーザヒアリング: LOD200はプロシージャル生成*
- **REQ-209**: 機器にmodelSrc（GLB URL）が指定されている場合、システムは外部3Dモデルを読み込んで表示しなければならない 🔵 *設計文書 §2.2 HvacEquipmentBase.modelSrc*

#### 制約要件

- **REQ-210**: 機器カタログ（メーカー別製品データ）およびN-BOM連携はMVPスコープ外とする 🔵 *ユーザヒアリング: 後回し*

### Phase 3-6: 後続リリース（スコープ定義のみ）

#### ダクトルーティング（Phase 3）

- **REQ-301**: システムはDuctSegmentRenderer（矩形/円形ダクトの3D生成）を提供しなければならない 🔵 *設計文書 §6 Phase 3*
- **REQ-302**: システムはDuctRouteTool（ポイント→ポイントのルーティング操作）を提供しなければならない 🔵 *設計文書 §4.3*
- **REQ-303**: ダクト寸法は等速法/等圧法で自動選定しなければならない 🔵 *設計文書 §3.3*
- **REQ-304**: 始点・終点指定で経路を自動提案し、ユーザーが調整できる半自動ルーティングを提供しなければならない 🔵 *ユーザヒアリング: 半自動*
- **REQ-305**: システムはダクト圧損をリアルタイムで計算・表示しなければならない 🔵 *設計文書 §3.4*

#### 配管ルーティング（Phase 4）

- **REQ-401**: システムは冷温水/冷媒/ドレン配管のルーティングツールを提供しなければならない 🔵 *設計文書 §6 Phase 4*
- **REQ-402**: 配管口径は自動選定し、圧損計算を実行しなければならない 🔵 *設計文書 §3.5*
- **REQ-403**: ポンプ揚程（配管抵抗+機器抵抗+高低差）を自動算出しなければならない 🔵 *設計文書 §3.5*

#### 検証＋拾い（Phase 5）

- **REQ-501**: システムは数量拾い出しエンジンを提供し、ダクト面積(m²)、配管長さ(m)、機器台数を自動集計しなければならない 🔵 *設計文書 §3.6*
- **REQ-502**: 積算出力はCSV、Excel（みつもりくん取込形式）、Rebroデータリンク形式に対応しなければならない 🔵 *設計文書 §5.3, ユーザヒアリング: 仕様確定済み*
- **REQ-503**: 警告システム（風速超過、圧損超過、未接続ポート）を提供しなければならない 🔵 *設計文書 §6 Phase 5*

#### 出口戦略（Phase 6）

- **REQ-601**: システムはIFC出力（空調設備→IFCエンティティ変換）を提供しなければならない 🔵 *設計文書 §5.2*
- **REQ-602**: IFC出力はSupabase Edge Functionsで実行しなければならない 🔵 *ユーザヒアリング*
- **REQ-603**: Rebro向けIFC最適化と同期ID埋め込みを提供しなければならない 🔵 *設計文書 §6 Phase 6*

## 非機能要件

### パフォーマンス

- **NFR-001**: 3Dビューポートは60fps以上で描画しなければならない（1000ノード以下の場合） 🟡 *既存Pascal Editorの性能目標から推測*
- **NFR-002**: 空調負荷概算の計算は100ゾーン以下で1秒以内に完了しなければならない 🟡 *m²単価法の計算量から推測*
- **NFR-003**: IFCファイルの読込は100MB以下のファイルに対応しなければならない 🟡 *web-ifc WASMの一般的な制限から推測*
- **NFR-004**: ダクト圧損計算はリアルタイム（ルーティング操作中）で更新しなければならない 🔵 *設計文書 §4.3*

### セキュリティ

- **NFR-101**: 認証にはbetter-authを使用し、未認証ユーザーのアクセスを拒否しなければならない 🔵 *既存実装・ユーザヒアリング*
- **NFR-102**: プロジェクトデータはSupabase RLSでユーザー単位のアクセス制御を行わなければならない 🟡 *既存Supabase構成から推測*

### ユーザビリティ

- **NFR-201**: UIは日本語で提供しなければならない（社内ツール） 🔵 *ユーザヒアリング: 社内設備設計者*
- **NFR-202**: 空調専門用語（AHU, PAC, FCU, VRF等）はそのまま使用してよい 🔵 *ユーザヒアリング: 設備設計者向け*
- **NFR-203**: フェーズ切替（ゾーニング→機器→ルート→計算→拾い）はワンクリックで行えなければならない 🔵 *設計文書 §4.1*

### 互換性

- **NFR-301**: 最新のChrome/Edgeブラウザ（WebGPU対応）で動作しなければならない 🔵 *既存技術スタック（WebGPU）*
- **NFR-302**: IFC 2x3 および IFC 4 形式の読込に対応しなければならない 🟡 *web-ifcの対応形式から推測*

### 拡張性

- **NFR-401**: 負荷計算エンジンは将来的にピーク負荷計算（時刻別計算）に拡張可能な設計としなければならない 🔵 *ユーザヒアリング: 将来詳細化*
- **NFR-402**: 機器スキーマはメーカーカタログ連携（N-BOM）に対応できるよう `definitionId` フィールドを持たなければならない 🔵 *設計文書 §2.2, ユーザヒアリング: 後回しだがスキーマは準備*
- **NFR-403**: マルチテナント拡張を将来考慮し、プロジェクトデータにテナントIDを付与できる余地を残さなければならない 🟡 *ユーザヒアリング: 後で検討*

## Edgeケース

### エラー処理

- **EDGE-001**: IFCファイルが不正またはパース不可能な場合、システムはエラーメッセージを表示し、部分的に読み込める要素は表示しなければならない 🟡 *web-ifcのエラーハンドリングから推測*
- **EDGE-002**: ゾーンポリゴンが自己交差している場合、システムは警告を表示し、修正を促さなければならない 🟡 *既存Zone実装のパターンから推測*
- **EDGE-003**: 機器のポートが未接続の状態で計算を実行した場合、システムは警告を表示しなければならない 🔵 *設計文書 §6 Phase 5*

### 境界値

- **EDGE-101**: ゾーン面積が0m²以下の場合、負荷計算はスキップしなければならない 🟡 *計算ロジックから推測*
- **EDGE-102**: ダクト風速が推奨値（主ダクト: 8m/s、枝ダクト: 6m/s）を超過した場合、警告を表示しなければならない 🔵 *設計文書 §3.3*
- **EDGE-103**: 配管口径が規格サイズに合致しない場合、最も近い規格サイズに丸めなければならない 🔵 *設計文書 §3.3 snapToStandard()*
